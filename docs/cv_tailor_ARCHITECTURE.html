What is cv-tailor?

A multi-agent decision-support system that automates the multi-model CV tailoring workflow. Takes a job description and a corpus of CV versions. Produces a tailored CV with a full audit trail.

Key implementation constraints

Import not subprocess

FastAPI imports tailor functions directly. No shell overhead, shared in-process state, debuggable stack traces. (RFI learning)

call_with_retry() everywhere

Wraps every API call from all three providers. Rate limits absorbed invisibly. Load-bearing from run 1, not polish.

asyncio.to_thread

All pipeline calls off the FastAPI event loop. Blocking I/O (ChromaDB, file writes, API calls) never stalls SSE delivery.

Filesystem-backed sessions

Per-run state in tmp/{run_id}/. TTL cleanup. Stateless containers — all persistence on bind-mounted host volumes.

ChromaDB cosine · immutable

Distance metric set at collection creation, cannot change. Metric mismatch gives wrong rankings silently. Verified on create.

nginx proxy_buffering off

Required for SSE streams to reach the browser without buffering delay. nginx default would break the live progress feed.

Section granularity — why sections, not CVs

Different convergence rates

Profile may converge after iteration 1. Experience at Microsoft may need three passes. Treating them as one unit wastes iterations on sections that are already done.

Cross-CV composition

Phase 1 picks the best source section per section_type across all CVs. The AI-focused CV's Skills section may be best even when the generic CV is the overall fit.

Intermediate version recovery

Every section version is written to disk. Phase 4 can surface that profile_v2 was sharper than profile_v3. Not possible with monolithic CV drafts.

Cost efficiency

Frozen sections are excluded from subsequent dual-write calls. By iteration 2–3 most sections are frozen. Later iterations are cheap regardless of max_iterations.

HITL #1 — Fit assessment

Conversational + Haiku-interpreted

cv-tailor presents section mix table, fit %, gaps. Human replies in free text ("swap Skills to generic CV"). Haiku interprets → structured decision shown back before pipeline resumes. no_fit stops pipeline with explanation; human can override.

HITL #2 — Section review

Lettered options + escape hatch

Section status table (converged / active / static). Unresolved items listed. Options [a–d] for known actions; [e] for free text. Only [e] triggers Haiku interpretation. Sonnet executes if revision needed.

HITL #3 — Formatting

Binary — Approve / Reject only

Haiku shows formatting diff + grounding report (Grounded Coverage %, Unsupported Claims count). Two buttons only. No Haiku interpretation needed — binary maps directly to pipeline state.

Design principles

The nine load-bearing decisions. Each one has a specific reason. "It seemed like a good idea" is not in any of them.

D-01 · D-02
Deterministic scaffold, bounded agentic loop. LLMs called as tools.
Phases 0–2 and 4–6 always run in order because the task is known. Phase 3 is agentic because the right number of iterations, which draft to select, and when to stop are genuinely unknown in advance. The model is not the orchestrator's concern — it receives a structured result from a tool that happens to call a model internally. This makes providers swappable without touching the orchestration loop.
D-04 · D-05
Dynamic scoring rubric with a JD-validation guard. Dual-signal convergence as the termination condition.
A static rubric measures the CV against the JD as written. A dynamic rubric measures it against what the JD means. The cap (max 2 additions/iteration) and JD-validation guard exist because F-28 proved they are load-bearing: Sonnet proposed 30+ rubric additions per run; the guard rejected every one. Without it, rubric inflation would have made the keyword coverage score meaningless. Two convergence signals (keyword_delta + quality_delta) because a single signal can be gamed; the F-28 Sonnet run confirmed the aggregate plateau path works even when the per-section freeze path contributes nothing.
D-06
Audit trail is separate from context. Reasoning is logged, never injected back.
Two independent concerns: context (what the model sees) and audit trail (what the human reads). Injecting accumulated reasoning verbosity back into the context inflates token cost and can distort future model behaviour. Keeping them separate means the context stays clean and the audit trail stays complete. Every orchestrator decision — accept/reject, rubric update, convergence — is logged to run_log.jsonl and readable after the run without reconstructing the conversation.
D-12 · D-13 · D-32
Section granularity. Static sections. Structural facts never enter draftable text.
The unit of work is a section, not a CV, because different sections converge at different rates. Interests and education are static — present in assembly, invisible to the loop. Job titles and dates are structural facts: F-29 showed that asking the drafter to preserve them produces inconsistent results (Microsoft lost its job title). The fix was structural: split them out before drafting, store in the manifest, re-attach verbatim at assembly. The model is a judgment engine, not a transcription service.
D-17
Phase 1 recommends a section-level mix across CVs, not a single base CV.
With section-granular ingestion, the best source for Profile may be a different CV than the best source for Skills. A single-CV recommendation leaves value on the table that the corpus already contains. The retrieval granularity should match the recommendation granularity. Keyword coverage — not semantic similarity — is the right signal for cross-variant selection because semantic scores barely differ across versions of the same person's section (F-11).
D-18
Conversational HITL with Haiku-interpreted free text. Interpretation shown before pipeline resumes.
Human review is the strongest safety mechanism in the system. The fit assessment checkpoint is conversational because the human has nuanced knowledge the model doesn't (Microsoft is PM work, not presales). The formatting checkpoint is binary because there's nothing to discuss. Haiku interprets free-text responses into structured decisions; the interpretation is shown back to the human before the pipeline resumes — so the pipeline always receives structured input and the human always knows what the system understood.
D-28
Dual-writer loop. Two independent drafters, one orchestrator, one pushback exchange.
The original design had GPT as a critic returning CritiqueItems. The actual best workflow was: draft with Claude, rewrite with GPT, arbitrate. Two writers with different training priors (Claude more diplomatic, GPT more direct) produce a richer output space than one writer with one critic. The orchestrator can synthesise — F-28 showed synthesis at 100% in iteration 1 dropping to 50% in iteration 2 as drafts converged, which is what "earned, not reflexive" synthesis looks like. Pushback is one exchange only to prevent the mechanism from becoming a token-burning negotiation.
D-33
CVCM: a durable candidate artifact consumed but never generated by the system.
Shifting from keyword optimisation toward value articulation produces more compelling CVs and more fabrication pressure. Better narrative context asks the model to frame experience through value creation patterns — which creates genuine pressure to invent claims the candidate can't support. The grounding check (Grounded Coverage % + Unsupported Claims count) is the mechanism that makes the CVCM safe to use. Authenticity and grounding are in tension; both must be tracked.
D-38 · D-39 · D-40
Not full auth — a spend guard. One capability cookie; fail closed; visibility via filtering not 403.
The threat model is: stop anyone but the owner from running expensive operations or mutating corpus state on a public portfolio deployment. Not multi-tenant access control. A passphrase-unlock issuing a signed HttpOnly cookie achieves the first goal without building accounts, roles, or sessions. One unlock authorises full mode runs and corpus writes — same threat, same guard. No FULL_MODE_KEY environment variable means demo-only, view-only corpus — fail closed by default. Run visibility uses filtering (404 on private runs, not 403) so private run IDs are not revealed to public visitors.
